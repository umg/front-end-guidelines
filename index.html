General

  1. Build everything with a progressive enhancement approach
  2. Think about how best to describe the information with code rather than just how to reproduce the design in a web browser
  3. Build mobile first and responsive, in most cases (info)
  4. The HTML5 Boilerplate is a great starting point for most new projects. Be sure to remove what isnâ€™t being used though
  5. Remember the separation between content (HTML) presentation (CSS) and behaviour (JavaScript)
  6. JavaScript should only progressively enhance the experience
  7. Code readability is preferable over file-size savings. There is no need to purposefully compress HTML or CSS whilst writing it; this should be part of the build process
  8. Always comment code where appropriate (example)

Markup

  1. Markup should be well-formed and semantically correct
  2. HTML5 Doctype and features should be used wherever appropriate
  3. Indentation should be 4 spaces. Correct indentation should be enforced to prevent unnecessary HTML comments and to improve maintainability
  4. The W3C Validator can be used to check for code errors, but 100% validation is not required
  5. Use Microformats where appropriate, specifically hCard and adr
  6. Always use title-case for headers and titles. Do not use all caps or all lowercase titles in markup, instead apply the CSS property text-transform:uppercase/lowercase
  7. Do not self-close HTML5 elements (e.g. use <br> instead of <br />)
  8. Ensure old XHTML code isn't used (example)

CSS

  1. Write CSS in LESS for all the benefits it brings
  2. Consider using * { box-sizing: border-box } on new responsive projects (info)
  3. Reference CSS files in the document before any JavaScript (example)
  4. Use the H5BP's stylesheet to normalise cross-browser inconsistencies
  5. Use classes for styling and JavaScript targeting purposes. Use IDs only where absolutely required (info)
  6. Know when to Float, and when to Position (info)
  7. Separate selectors and declarations by new lines (example)
  8. Use meaningful or generic class names (example)
  9. Class names should be lowercase and separated with dashes (info). Use camelCase for JS
  10. Understand CSS specificity to write most efficient selectors and to avoid over specifying
  11. Don't use !important to override rules. This is not needed with correct CSS specificity
  12. Write CSS properties as shorthand wherever possible (info)
  13. Omit unit specification after "0" values (info)
  14. Omit quotation marks in URI values (info)
  15. Use @font-face for embedding custom fonts (generator) and be aware of performance implications

JavaScript

  • Place all JavaScript at the end of your document, except in special circumstances
  • Use single quotes as much as possible (info)
  • Feature sniff, rather than browser sniff, using Modernizr (info)
  • Don't use document.write() (info). There are better alternatives such as .innerHTML or .createElement
  • Use camelCase for variable and function names (info)
  • Constants or configuration variables (like animation durations, etc.) should be at the top of the file
  • Comment your code effectively and make it easy to read. Minification happens later
  • Place all global plugins within plugins.js and all global JavaScript within main.js

Accessibility

  1. Understand and use ARIA (intro)
  2. Interfaces should be checked against W3C's Checklist for Web Content Accessibility
  3. Interfaces should meet WCAG Priority 1 Guidelines
  4. More accessibility tools

Performance

  1. All CSS and JavaScript files should be minified and in as few files as is sensible for deployment to live servers
  2. Google's Web Performance Best Practices should be adhered to
  3. Images should be compressed with Smush.it (PNG) JPEGmini (JPEG) for deployment to live servers
  4. Leverage CSS sprites where appropriate to reduce page weight and HTTP requests
